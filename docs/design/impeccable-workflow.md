# Impeccable 设计工作流

这份文档定义了当前项目如何使用 `impeccable` skill，也整理了一套可迁移到其他项目的 UI 优化工作流。

它不是“随便挑几个 skill 用一下”的清单，而是一套固定顺序、固定判断标准、固定收尾方式的工作方法。

## 1. 全局硬规则

以下规则适用于本项目所有页面，也适合作为其他项目的基础规则。

### 1. 页面必须占满整个屏幕
- 每一页都要形成完整的全屏工作界面
- 页面要主动利用 viewport 高度，而不是只堆一段内容
- 内容不够时要通过布局组织来撑起页面，不允许留下无意义的大块空白

### 2. 页面必须对不同屏幕稳定适配
- 浏览器窗口变化、不同屏幕尺寸、不同缩放比例下都要保持整齐
- 不允许出现排版错乱、乱码、位置偏移、控件重叠、模块溢出
- 响应式适配必须以“内容重排”为目标，而不是被动压缩

### 3. 页面必须按内容选择最佳呈现形式
- 短操作区：优先工具条、固定区、贴边控制
- 长列表：优先垂直滚动区域、分组或分页
- 图像卡片：优先网格、滑轨、对比布局
- 双栏工作台：优先分离操作区和结果区，必要时只让一侧滚动
- 长表单：优先步骤化、分段化、折叠化

### 4. 滚动策略必须由内容决定
- 不默认整页滚动
- 决定使用整页滚动、局部滚动、横向滑动、弹窗、抽屉还是固定面板时，必须基于模块职责和内容长度
- 滚动和固定逻辑必须帮助用户理解和操作，而不是制造额外负担

### 5. 系统一致性是基本要求
- 同层级字体风格必须一致
- 同角色按钮样式必须一致
- 同类动画时长、节奏、反馈方式必须一致
- 不同页面之间的组件风格必须可以互相迁移，而不是每页一套语言

## 2. 先做什么：项目初始化

### 第一步：跑 `teach-impeccable`
在项目开始或设计方向明显变化时，先跑一次 `teach-impeccable`。

输出结果应该包括：
- `.impeccable.md`
- 用户是谁
- 产品气质是什么
- 明确不要像什么
- 页面占屏、响应式、内容驱动展示这些全局规则

### 第二步：建立页面级 workflow
项目初始化完成后，再为每个核心页面定义固定工作流，避免每次优化都从零开始判断。

## 3. 当前项目的页面级工作流

### 首页
顺序：
`critique -> arrange -> typeset -> polish -> audit`

关注重点：
- 首页是否真正像“控制台”而不是空信息页
- 是否占满屏幕并建立清晰的上下结构
- 数据概览、快捷入口、最近变化是否一眼有主次
- 不同窗口尺寸下是否仍然能稳定展示所有板块

收尾检查：
- 顶部、主体、下方区域是否形成完整节奏
- 数据卡和最近模块是否出现错位或拥挤
- 是否仍然有 AI 模板感

### 衣柜页
顺序：
`critique -> arrange -> clarify -> adapt -> polish`

关注重点：
- 高密度信息是否依然容易扫读
- 筛选、搜索、卡片或表格之间的关系是否清晰
- 是否占满屏幕并合理利用垂直空间
- 窄屏、缩放、切换视图时是否保持稳定

收尾检查：
- 控制区是否清楚
- 浏览区是否顺手
- 列表/卡片是否存在密度失衡或模块漂移

### 添加衣服页
顺序：
`clarify -> harden -> adapt -> polish`

关注重点：
- 用户是否一眼知道该做什么
- AI 处理中每一步是否被清楚解释
- 空状态、加载态、错误态是否完整
- 表单、上传区、结果区在不同尺寸下是否依然稳定

收尾检查：
- 是否减少了输入负担
- 是否有明确的下一步
- 是否避免长页面失控和局部模块挤压

### 穿搭页
顺序：
`critique -> arrange -> adapt -> polish`

关注重点：
- 左右双栏是否真正解耦
- 预览区是否足够大且稳定
- 衣物选择区是否合理利用全屏高度
- 是否根据内容正确选择局部滚动、横向滑动、固定区、弹窗等呈现方式

收尾检查：
- 左右关系是否清楚
- 局部滚动是否符合直觉
- 控制区、预览区、历史区是否在不同尺寸下仍然整齐

### 整站上线前
顺序：
`audit -> harden -> polish`

关注重点：
- 全站是否仍然保持统一的字体、按钮、间距、动画语言
- 是否存在响应式问题、可访问性问题、边界状态缺失
- 是否还有页面没有占满屏幕或在某些尺寸下崩掉

收尾检查：
- 全站一致性
- 边界状态完整性
- 发布前风险项

## 4. 可迁移到其他项目的通用工作流

这部分是你以后在其他项目里也能复用的模板。

### 模板 A：首页 / 仪表盘 / 控制台
顺序：
`critique -> arrange -> typeset -> polish -> audit`

适合：
- 产品首页
- 仪表盘
- 总览页

核心目标：
- 建立主次层级
- 让内容占满屏幕并形成完整节奏
- 让用户快速知道“我现在能做什么”

### 模板 B：高密度管理页 / 列表页 / 资源页
顺序：
`critique -> arrange -> clarify -> adapt -> polish`

适合：
- 管理页
- 资源库
- 商品/素材/文件列表页

核心目标：
- 控制信息密度
- 保持筛选和浏览效率
- 在不同宽度下都稳定可用

### 模板 C：表单页 / 上传页 / 配置页
顺序：
`clarify -> harden -> adapt -> polish`

适合：
- 表单
- 上传流程
- 创建流程
- 设置页

核心目标：
- 降低理解成本
- 补齐状态反馈
- 让长流程按内容选择最合适的分段呈现方式

### 模板 D：工作台 / 编辑器 / 画布页
顺序：
`critique -> arrange -> adapt -> polish`

适合：
- 编辑器
- 预览页
- 左右双栏工作台
- 生成式工具页

核心目标：
- 平衡操作区和结果区
- 按职责决定谁固定、谁滚动、谁弹出
- 让复杂界面在多种尺寸下仍然稳定

### 模板 E：发版前收尾
顺序：
`audit -> harden -> polish`

适合：
- 上线前总检查
- 阶段性收尾

核心目标：
- 查出排版、响应式、边界状态和一致性问题
- 做最后一轮系统级收口

## 5. 具体怎么用：标准优化步骤

无论是本项目还是其他项目，建议按下面 8 步执行。

### 第 1 步：先建上下文
运行 `teach-impeccable`，明确：
- 用户是谁
- 产品是干什么的
- 品牌气质是什么
- 不要像什么
- 是否需要深色模式
- 页面是否必须全屏占满
- 哪些模块更适合固定、滚动、滑动、弹窗

### 第 2 步：拆页面模块
不要一上来改样式，先把页面拆成模块：
- 导航区
- 概览区
- 控制区
- 内容区
- 历史区
- 结果区
- 表单区

然后逐个判断每个模块的职责、长度、密度和优先级。

### 第 3 步：先诊断，再动手
如果问题不明确，先跑：
- `critique`：看体验和层级问题
- `audit`：看技术质量、响应式、可访问性和反模式问题

先知道问题在哪，再决定后续 skill 顺序。

### 第 4 步：先决定展示方式
这一步很关键。

在开始改布局前，必须先判断：
- 这个模块该固定还是滚动
- 该横向滑动还是纵向展开
- 该做弹窗还是内嵌面板
- 该双栏还是单栏
- 是否要分段、折叠、分页、步骤化

也就是：先决定“内容如何呈现”，再决定“样式怎么美化”。

### 第 5 步：按页面类型跑主工作流
根据页面类型套用对应 workflow：
- 首页：`critique -> arrange -> typeset -> polish -> audit`
- 衣柜页：`critique -> arrange -> clarify -> adapt -> polish`
- 添加衣服页：`clarify -> harden -> adapt -> polish`
- 穿搭页：`critique -> arrange -> adapt -> polish`

其他项目则套用最接近的通用模板。

### 第 6 步：单独做响应式与占屏检查
每一轮改完后，都必须单独检查这 4 件事：
- 页面是否占满屏幕
- 浏览器缩放后是否稳定
- 不同宽度下是否仍然整齐
- 模块是否用了正确的滚动/滑动/固定方式

这一步不能省，因为很多“看起来小”的问题最后都会变成整页崩坏。

### 第 7 步：最后才做 polish
`polish` 应该放在结构稳定之后。

它主要解决：
- 对齐
- 间距
- 小状态
- 小动画
- 微文案一致性
- 同一套组件在不同页面中的统一感

### 第 8 步：发版前做总审计
上线前统一跑：
`audit -> harden -> polish`

把这轮当作发布门槛，而不是可选项。

## 6. 可直接复用的提示词模板

### 诊断问题
- `用 critique 看一下首页，重点看层级、认知负担、占屏完整性和 AI 味。`
- `用 audit 检查穿搭页，重点看响应式、局部滚动策略和可访问性。`

### 调整结构
- `用 arrange 优化衣柜页的筛选区和内容区关系，要求页面占满屏幕。`
- `用 arrange 调整穿搭页左右结构，并根据内容决定哪一栏固定、哪一栏滚动。`

### 调整文案与流程
- `用 clarify 优化添加衣服页的提示文案和操作命名。`
- `用 harden 补齐添加衣服页的空状态、加载态、错误态。`

### 响应式适配
- `用 adapt 优化首页在不同窗口尺寸下的排版稳定性。`
- `用 adapt 检查衣柜页在窄屏和缩放后是否仍然完整占屏。`

### 收尾
- `用 polish 做一轮穿搭页收尾，重点看按钮、标题、间距和动画统一性。`
- `用 audit -> harden -> polish 做一轮整站上线前检查。`

## 7. 每次优化的产出标准

每一次 workflow 最好都输出这几项：
- 本轮发现的核心问题
- 为什么用这个 skill 顺序
- 哪些模块被改，哪些刻意不改
- 页面是否占满屏幕、是否通过响应式检查
- 还有哪些剩余风险

这样这套流程既能指导实现，也能沉淀成复盘资料。
